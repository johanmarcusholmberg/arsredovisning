## 1. Core goal

Implement a robust note management layer where users can:

- see which notes are required
- see which notes are suggested
- see which notes are optional
- select or deselect optional notes
- confirm required notes
- edit note text where allowed
- review note tables
- inspect source accounts behind note rows
- see exactly which balance sheet or income statement line each note belongs to
- see exactly which note number is shown in the financial statements
- validate that note totals match report lines
- prevent inconsistent note references before preview/export

This phase should make the note system reliable enough for future export.

## 2. Note classification

Each note should have a clear status/classification.

Support at least:

- required
- suggested
- optional
- not_applicable
- hidden
- manually_added

Definitions:

### required

A note required by the selected reporting framework or triggered by imported/mapped data.

Required notes cannot be removed if they are legally or structurally needed.

The user may confirm or edit allowed parts, but cannot disable the note unless the underlying trigger no longer applies.

### suggested

A note that the system recommends based on mapped data, account types or report structure.

The user may accept, ignore or mark as not applicable, depending on rules.

### optional

A note that the user may include for better explanation, even if not strictly required.

The user may include or exclude optional notes.

### not_applicable

A note that the system considered but the user or validation logic marked as not applicable.

It must not appear in the formal annual report.

It must not be referenced from the balance sheet or income statement.

### hidden

A note used internally or removed from the formal annual report.

Hidden notes must not be included in automatic note numbering.

Hidden notes must not be referenced in financial statement tables.

### manually_added

A note created by the user.

Manual notes must still have:
- note number if included
- title
- body or table
- linked statement line if relevant
- validation status
- source/traceability where relevant

## 4. Automatic note numbering

Implement automatic note numbering for all included notes.

Rules:

- note numbers must be unique
- note numbers must be sequential
- hidden notes must not receive visible note numbers
- not_applicable notes must not receive visible note numbers
- excluded optional notes must not receive visible note numbers
- manually added included notes must receive a valid number
- numbering should follow the order of appearance in the annual report
- notes referenced from resultaträkning and balansräkning must use the same number shown in the notes section
- if note order changes, references must update automatically
- note numbers must never be manually typed into financial statement lines as disconnected text

The note number source of truth should be one central function or module.

Do not allow separate note numbering logic in:
- report tables
- note section
- preview
- future export

Later Phase 6.6 and Phase 7 must be able to reuse this same note numbering source.

## 5. Note references in financial statements

Resultaträkning and Balansräkning must support a “Not” column.

In this phase, prepare or implement the data structure and UI logic for this column.

For each financial statement line:

- show the linked note number if a note is included
- show blank if no note applies
- do not show notes marked hidden, excluded or not_applicable
- support multiple note references if needed
- references must point to actual note IDs, not only text labels
- references must update when note numbering changes
- references must be consistent across review, preview and later export

Each financial statement line should support:

- statementLineId
- label
- amountCurrentYear
- amountPreviousYear
- linkedNoteIds
- visibleNoteNumbers
- sourceAccountIds
- validationStatus

The UI should make it clear which note belongs to which row.

Example:

In balance sheet review:

| Rad | Not | 2026 | 2025 |
|---|---:|---:|---:|
| Kundfordringar | 12 | 250 000 | 210 000 |

Clicking note number 12 should open the linked note.

In the note view, clicking “Visa kopplad rad” should navigate back to the corresponding statement line.

## 6. Note totals and reconciliation

Implement note total reconciliation.

For each note linked to a financial statement line, validate:

note total = linked statement line amount

When previous-year values are available:

previous-year note total = previous-year linked statement line amount

If there is a mismatch, show:

- affected note
- affected statement line
- note amount
- statement line amount
- difference
- likely reason
- recommended action

The system should support structured validation results.

Suggested validation result fields:

- status:
  - passed
  - warning
  - blocking_error
- code
- message
- affectedNoteId
- affectedStatementLineId
- currentYearDifference
- previousYearDifference
- severity
- recommendedAction

In this phase, mismatches should be clearly visible and structured for later blocking in Phase 7.

Do not allow a misleading “green” status when totals do not match.

## 13. Validation

Create reusable note validation functions.

At minimum, validate:

- included notes have unique note numbers
- included notes have sequential note numbers
- financial statement note references point to included notes
- no hidden note is referenced
- no not_applicable note is referenced
- no excluded optional note is referenced
- required notes are included unless clearly not applicable by rule

- previous-year note totals match previous-year statement lines when applicable
- manual notes have valid title/content
- manual overrides have required reason/comment
- system-generated note text requiring confirmation is confirmed before export readiness

Validation should return structured results.

Suggested result:

- status
- severity
- code
- message
- affectedNoteId
- affectedNoteRowId
- affectedStatementLineId
- recommendedAction

This phase does not need to implement final export blocking, but the validation result structure must support blocking in Phase 7.
