## PART 2 — Real AI-assisted mapping and explanations

### Problem

The website currently says or implies that the product has AI help, but I am not sure whether there is actual useful AI assistance in the system when it comes to:

- Account mapping
- Explaining why accounts belong on certain rows
- Suggesting corrections
- Detecting incorrect classification
- Helping the user make safe and correct changes

I want the product to feel like the user is actively getting help, not just seeing static mapping tables.

### Goal

Implement a practical AI-assistance layer for account mapping and annual-report classification.

This does not need to be a perfect external AI integration yet if no AI provider is configured, but the system should have a real architecture and useful behavior from day one.

The user should feel guided when mapping accounts and resolving classification issues.

---

## Required features

### 1. Mapping assistant panel

Add an assistant/help panel in the account mapping review step.

The panel should help the user understand:

- What account is being mapped
- Which annual-report row it currently maps to
- Why that row is suggested
- What source accounts contribute to the row
- Whether the mapping looks normal, unusual or risky
- What alternative rows may be relevant
- What effect a change would have on the balance sheet, income statement, notes and validation

Example UI labels:

- “AI mapping help”
- “Suggested classification”
- “Why this row?”
- “Possible issue”
- “Recommended action”
- “Show calculation”
- “Apply suggestion”
- “Dismiss”

Use Swedish user-facing copy in the app.

### 2. Rule-based AI fallback

If no external AI provider is configured, build a deterministic rule-based assistant that behaves like an AI helper.

This fallback should use:

- BAS account number ranges
- Existing mapping rules
- Imported balances
- Debit/credit sign
- Current financial statement row
- Existing mapped row
- Whether the same account type appears on both asset and liability side
- Framework selected: K2/K3/RFR2/IFRS, if available
- Previous-year values, if available

The code should be structured so that an external AI provider can later be connected without rewriting the UI.

Create a clean service layer, for example:

- `mappingAssistantService`
- `classificationSuggestionService`
- or similar existing naming pattern

It should return structured suggestions, not raw text only.

Suggested return shape:

```ts
type MappingAssistantSuggestion = {
  accountId: string;
  accountNumber: string;
  accountName: string;
  currentRowId: string | null;
  suggestedRowId: string | null;
  confidence: "low" | "medium" | "high";
  severity: "info" | "warning" | "blocking";
  reason: string;
  explanation: string;
  recommendedAction: "keep" | "remap" | "net" | "review" | "manual_adjustment";
  calculationPreview?: {
    before: number;
    after: number;
    affectedRows: Array<{
      rowId: string;
      rowLabel: string;
      before: number;
      after: number;
    }>;
  };
  source: "rules" | "ai";
};

Adapt the exact type to the existing codebase.

3. Smart reclassification and netting guidance

Add intelligent handling for situations where values appear on both asset and liability sections but should be presented netted or reclassified according to annual-report logic.

Important example:

If a value is currently shown under receivables but, based on sign/current balance/classification logic, it should instead reduce or move to liabilities — or the other way around — the system should not simply show duplicated or contradictory rows.

The assistant should detect cases such as:

Same counterparty/account group appears on both receivable and liability side
Negative receivable balances
Negative liability balances
Account groups that should normally be netted or reclassified for presentation
Imported balances that produce values in both sections where the annual-report row should show a net amount
Current and previous-year values pointing in different directions

The system should suggest a correction such as:

Move amount from receivable row to liability row
Move amount from liability row to receivable row
Net two related rows into the correct annual-report presentation row
Keep separate, but require user review/comment
Require manual adjustment when the system cannot safely decide

The user must be able to review the suggestion before applying it.

Do not silently change financial statement rows without showing the user.

4. Example behavior for receivables vs liabilities

Implement logic for cases similar to this:

A mapped account or account group creates a value under receivables
Another related account or sign-adjusted value creates a value under liabilities
The two values should not appear incorrectly as separate positive values if presentation logic requires netting or reclassification
The assistant should calculate the net effect and propose the correct row

Example user-facing explanation:

“Det finns värden både på fordrings- och skuldsidan för denna grupp. Systemet rekommenderar att beloppen nettas och presenteras på den sida där nettot hör hemma. Granska beräkningen innan du tillämpar ändringen.”

The calculation preview should show:

Current receivable row amount
Current liability row amount
Net amount
Proposed final row
Affected annual-report row(s)
Whether notes are affected

7. Validation integration

The mapping assistant should connect to validation.

If the assistant detects risky classification issues that the user has not resolved, validation should show warnings.

Examples:

“Möjlig felklassificering mellan fordringar och skulder”
“Negativt saldo på rad som normalt inte ska vara negativ”
“Belopp finns både på tillgångs- och skuldsidan och bör granskas”
“Manuell kommentar krävs för att behålla nuvarande klassificering”

Some issues can be warnings, others can be blocking only if they would make the annual report structurally inconsistent.

Do not block export for every uncertain mapping issue, but high-risk unresolved issues should require review/comment before export.
