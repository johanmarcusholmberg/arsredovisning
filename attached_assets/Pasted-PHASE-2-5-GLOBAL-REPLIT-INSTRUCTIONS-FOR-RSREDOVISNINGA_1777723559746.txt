Continue working on “Årsredovisningar”.

This is a backend/security/storage preparation phase before real import and export functionality.

Do not build SIE/Excel/CSV import parsing yet.
Do not build financial statement generation yet.
Do not build notes, validation or export yet.

Your task is to create a secure backend foundation for a financial-data application using Supabase.

This project handles sensitive accounting and annual-report data. Security, file storage, project access, audit logging and Row Level Security must be designed before import/export features are built.

Core product rules:
- Free users can only access the demo workspace.
- Free users cannot upload real files.
- Real projects require a paid project entitlement.
- Payment unlocks one real annual-report project for one company and one financial year.
- Users must only access projects they own or have been invited to.
- Demo data and real customer data must never be mixed.
- Annual report output is Swedish only, but website/app UI supports Swedish and English.

Build the following:

1. Backend architecture review

Review the existing backend structure and create a clean architecture for:

- Supabase database
- Supabase Auth
- Supabase Storage
- server-side actions/API routes/edge functions
- frontend data access helpers
- permission helpers
- audit logging helpers

Document where backend logic should live.

Rules:
- Sensitive authorization checks must not rely only on frontend logic.
- File upload and export access must be checked server-side.
- Project access must always be checked through database policies or server-side helpers.
- Do not expose service-role keys to the browser.

4. Row Level Security

Enable RLS for all customer/project-related tables.

Create policies so that:

profiles:
- users can read/update their own profile only

companies:
- users can read companies where they have project access or are owner
- only owner/admin can update company metadata

annual_report_projects:
- users can read projects where they have project_access
- only users with owner/admin/accountant role can update project data
- free users cannot create real projects unless they have an active entitlement

project_access:
- users can read access rows for projects they belong to
- only owner/admin can invite/revoke access

project_entitlements:
- users can read their own entitlements
- entitlement creation/update should be server-side only

audit_events:
- users can read audit events for projects they can access
- audit event insertion should be through server-side helper or controlled function

project_snapshots:
- users can read snapshots for projects they can access
- only owner/admin/accountant can create snapshots

Important:
Do not rely on hidden frontend buttons as the only security layer.

6. Supabase Storage buckets

Create/define secure storage buckets for:

- import-files
- previous-annual-reports
- cover-sheets
- exports
- demo-assets

Rules:
- import-files: private
- previous-annual-reports: private
- cover-sheets: private
- exports: private
- demo-assets: public or controlled read-only

Storage path convention:
import-files/{project_id}/{import_file_id}/{original_filename}
previous-annual-reports/{project_id}/{file_id}/{filename}
cover-sheets/{project_id}/{cover_sheet_id}/{filename}
exports/{project_id}/{export_id}/{filename}
demo-assets/{asset_name}

Rules:
- Users can only access files for projects they can access.
- Free users cannot upload to real project buckets.
- Demo assets must never share storage paths with real customer data.
- Exports should use signed URLs with expiration, not public permanent URLs.
- Imported files and exports should not be publicly accessible.

8. File upload security

Implement or prepare secure upload flow.

Requirements:
- Check user permission before allowing upload.
- Check project entitlement before allowing real import upload.
- Allow only approved file types:
  - SIE
  - CSV
  - XLS/XLSX
  - PDF for previous annual report/cover sheet
  - image formats for cover image if needed
- Validate file size.
- Store file metadata in database.
- Store file in private bucket.
- Log upload events.
- Show friendly errors.

Do not parse files yet in this phase.
Just create the secure upload/storage foundation.

9. File download/security

- Demo sample files can be public or read-only demo assets.
